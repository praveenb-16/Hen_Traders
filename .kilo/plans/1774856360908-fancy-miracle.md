# Plan: Separate Database Tables for Hen Types + Daily Balance

## Goal

Replace the single monolithic `Transaction` table (with 30+ nullable columns) with 4 clean, purpose-specific tables:
1. `KattiKoli` — Katti Koli transaction data
2. `NallaKoli` — Nalla Koli transaction data
3. `BothTransaction` — Combined Katti + Nalla transaction (standalone, all fields in one table)
4. `DailyBalance` — Daily balance/extra financial tracking (the "ledger" the user queries)

---

## New Prisma Schema

```prisma
model KattiKoli {
  id          String        @id @default(cuid())
  date        DateTime      @default(now())
  box1        Int
  hen1        Int
  box2        Int
  hen2        Int
  box3        Int?
  hen3        Int?
  totalHens   Int
  freeHen     Int?
  rate        Float
  amount      Float
  labour      Float
  total       Float
  dailyBalance DailyBalance @relation(fields: [dailyBalanceId], references: [id])
  dailyBalanceId String
  createdAt   DateTime      @default(now())
  @@index([date])
}

model NallaKoli {
  id          String        @id @default(cuid())
  date        DateTime      @default(now())
  box1        Int
  hen1        Int
  box2        Int
  hen2        Int
  box3        Int?
  hen3        Int?
  totalHens   Int
  freeHen     Int?
  netWeight   Float
  waterWeight Float
  weight      Float
  rate        Float
  amount      Float
  labour      Float
  total       Float
  dailyBalance DailyBalance @relation(fields: [dailyBalanceId], references: [id])
  dailyBalanceId String
  createdAt   DateTime      @default(now())
  @@index([date])
}

model BothTransaction {
  id          String        @id @default(cuid())
  date        DateTime      @default(now())

  // Katti Koli fields
  kBox1       Int
  kHen1       Int
  kBox2       Int
  kHen2       Int
  kBox3       Int?
  kHen3       Int?
  kTotalHens  Int
  kFreeHen    Int?
  kRate       Float
  kAmount     Float
  kLabour     Float
  kTotal      Float

  // Nalla Koli fields
  nBox1       Int
  nHen1       Int
  nBox2       Int
  nHen2       Int
  nBox3       Int?
  nHen3       Int?
  nTotalHens  Int
  nFreeHen    Int?
  nNetWeight  Float
  nWaterWeight Float
  nWeight     Float
  nRate       Float
  nAmount     Float
  totalAmount Float
  dailyBalance DailyBalance @relation(fields: [dailyBalanceId], references: [id])
  dailyBalanceId String
  createdAt   DateTime      @default(now())
  @@index([date])
}

model DailyBalance {
  id          String        @id @default(cuid())
  date        DateTime      @default(now())
  henType     HenType
  totalAmount Float
  paidAmount  Float
  todayAmount Float
  todayType   DiffType
  oldAmount   Float
  oldType     DiffType
  finalAmount Float
  finalType   DiffType
  kattiKoli   KattiKoli[]
  nallaKoli   NallaKoli[]
  bothTransaction BothTransaction?
  createdAt   DateTime      @default(now())
  @@index([date])
}

enum HenType {
  KATTI_KOLI
  NALLA_KOLI
  BOTH
}

enum DiffType {
  BALANCE
  EXTRA
}
```

## API Response Shape

The API will continue to return a flat object that matches what the frontend expects. The API route assembles the shape from the related records:

```json
{
  "id": "dailyBalanceId",
  "date": "...",
  "henType": "KATTI_KOLI",
  "kBox1": 5, "kHen1": 10, ...,
  "nBox1": null, ...,
  "totalAmount": 5000,
  "paidAmount": 4000,
  "todayAmount": 1000,
  "todayType": "BALANCE",
  "oldAmount": 500,
  "oldType": "BALANCE",
  "finalAmount": 1500,
  "finalType": "BALANCE"
}
```

This means **minimal frontend changes** — the existing `Transaction` interface in history/BillDialog continues to work.
